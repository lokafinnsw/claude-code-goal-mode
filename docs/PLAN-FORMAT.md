# Plan Format ‚Äî `tree.json` schema and `plan.md` ‚Üî tree round-trip

This is the 1.0.0 reference for the goal-mode plan format. It documents every
field of `tree.json`, the `plan.md` Markdown convention, and the rules under
which the two stay in sync.

For other reference docs see [BUDGET.md](./BUDGET.md),
[REVIEW-AGENTS.md](./REVIEW-AGENTS.md), and [ANTI-PATTERNS.md](./ANTI-PATTERNS.md).

---

## 1. Why two files?

A goal lives at `<project>/.claude/goals/active/` and is described by two
files that mirror each other:

- **`plan.md`** ‚Äî human-readable Markdown, edited by the user (and initially
  generated by `/goal:plan`). This is the file you read in your editor when
  you want to see what the goal is and what the structure looks like.
- **`tree.json`** ‚Äî machine-readable JSON, validated by zod on every load
  (`engine/state.mjs:180-188`). This is what the engine actually consumes.

Both are written when `/goal:plan` produces a plan. The user is encouraged to
edit `plan.md` only while `state.lifecycle` is `draft` or `approved` ‚Äî
`/goal:approve-plan` stamps `tree.approved_at` and freezes the structure.
After that point the plan is locked; the engine never modifies node
identities, criteria, or hierarchy at runtime, only the per-node `status`,
`evidence[]`, `review_attempts`, `notes[]`, and `blocker_reason`.

The round-trip rule is: **the two files must describe the same hierarchy
with the same ids, criteria, reviewers, and validate commands.**
`/goal:approve-plan` validates `tree.json` against the schema; matching it
against `plan.md` is the user's responsibility (and the prompt-bootstrap is
explicit about writing both ‚Äî see `prompts/plan-bootstrap.md`).

## 2. `tree.json` top-level fields

The full schema lives at `engine/state.mjs:59-66` (`GoalTreeSchema`).

| Field | Type | Required | Notes |
|---|---|---|---|
| `schema_version` | literal `1` | yes | `engine/state.mjs:60`. Pinned; future migrations live in `engine/state.mjs`. |
| `goal_id` | string (‚â•1 char) | yes | Slug, immutable for the lifetime of the goal. Convention: kebab-case (e.g. `mancelot-mvp-2026-05`). `engine/state.mjs:61`. |
| `mission` | string (‚â•1 char) | yes | One-line top-level objective. `engine/state.mjs:62`. |
| `created_at` | ISO-8601 datetime | yes | Stamped by `/goal:plan` when the plan is first written. `engine/state.mjs:63`. |
| `approved_at` | ISO-8601 datetime or `null` | yes | `null` until `/goal:approve-plan` stamps it (`engine/approve-plan-cli.mjs:94`). |
| `root` | `GoalNode` | yes | Recursive node tree (see ¬ß3). By convention the root is a `sprint` node, but the schema also accepts a single `task` (used by the smoke-test fixture in `docs/SMOKE-TEST.md:28-50`). `engine/state.mjs:65`. |

### Example (top-level)

```json

  "schema_version": 1,
  "goal_id": "smoke",
  "mission": "Touch a file",
  "created_at": "2026-05-09T00:00:00.000Z",
  "approved_at": "2026-05-09T00:00:00.000Z",
  "root": { ... }
}
```

## 3. `GoalNode` fields

The recursive schema is at `engine/state.mjs:29-57` (`GoalNodeObjectSchema`
and `GoalNodeSchema`). Every node ‚Äî sprint, epic, or task ‚Äî has the same
shape; the `type` field distinguishes them and the `children` field is empty
for tasks (leaf nodes).

| Field | Type | Notes |
|---|---|---|
| `id` | string (‚â•1 char) | Path-style: `sprint-1.epic-2.task-3`. Must be unique across the tree. `engine/state.mjs:30`. |
| `type` | `"sprint"` \| `"epic"` \| `"task"` | `engine/state.mjs:15` (`NodeTypeSchema`). |
| `title` | string (‚â•1 char) | Short human label. `engine/state.mjs:32`. |
| `goal` | string (‚â•1 char) | What "done" looks like, 1‚Äì3 sentences. `engine/state.mjs:33`. |
| `acceptance_criteria` | string[] | Each entry must be ‚â•1 char. **Tasks must have ‚â•1 criterion** ‚Äî enforced by the `.refine()` at `engine/state.mjs:54-57`. Sprint and epic nodes may have empty `[]` (criteria roll up from leaf tasks). |
| `review` | string[] | Opaque subagent_type names. The engine never interprets them ‚Äî see [REVIEW-AGENTS.md](./REVIEW-AGENTS.md). `engine/state.mjs:35`. |
| `validate` | string \| `null` | A shell command run by the agent for verification (e.g. `npm test -- src/foo`). `null` means no automation. `engine/state.mjs:36`. |
| `work_front` | string \| `null` | Free-form parallelism label, e.g. `"engine"`, `"art"`, `"narrative"`. `engine/state.mjs:37`. |
| `status` | `NodeStatusSchema` | One of `pending`, `pursuing`, `review-pending`, `achieved`, `blocked`, `skipped`. `engine/state.mjs:6-13`. Set to `pending` by `/goal:plan`. |
| `evidence` | `Evidence[]` | Accumulated across iterations by `applyMutations` (`engine/apply-mutations.mjs:120-140`). Empty `[]` at plan time. |
| `blocker_reason` | string \| `null` | Populated when status becomes `blocked` (`engine/apply-mutations.mjs:163`, `:214`). |
| `review_attempts` | int ‚â•0 | Per-node block counter. Incremented on `<task-status>blocked</task-status>` (`engine/apply-mutations.mjs:161`) and on any audit-verdict batch containing NOGO/REVISE (`engine/apply-mutations.mjs:211`). At ‚â•3 the node auto-blocks (`engine/apply-mutations.mjs:212-216`). |
| `notes` | string[] | Ephemeral per-turn agent notes. Accepts `[]` at plan time. |
| `children` | `GoalNode[]` | Recursive. Tasks have `[]`; sprints and epics have child nodes. Lazy reference at `engine/state.mjs:49`. |

### `Evidence` shape

The schema is at `engine/state.mjs:17-27`.

| Field | Type | Notes |
|---|---|---|
| `ts` | ISO-8601 datetime | Stamped by `applyMutations` (`engine/apply-mutations.mjs:122`). |
| `iteration` | int ‚â•0 | The iteration counter at the time evidence was recorded. |
| `criterion_index` | int ‚â•0 \| `null` | Index into the parent task's `acceptance_criteria[]`. Out-of-range indices are silently dropped from coverage tallying but the evidence record itself is still kept (`engine/apply-mutations.mjs:98-106`). |
| `file` | string \| `null` | Path-relative-to-repo. |
| `line` | int \| `null` | Line number. |
| `commit` | string \| `null` | Git sha. |
| `command` | string \| `null` | Shell command run. |
| `exit_code` | int \| `null` | Command exit code. |
| `note` | string | Free text ‚Äî what this evidence demonstrates. |

The agent populates evidence by emitting `<evidence ... />` tags; the parser
contract is documented at `engine/parse-tags.mjs:8-50`.

```json
  "evidence": [],
  "blocker_reason": null,
  "review_attempts": 0,
  "notes": [],
  "children": []
}
```

## 4. `plan.md` Markdown convention

`plan.md` is the human-readable mirror of the same hierarchy. The convention
matches the design spec ¬ß5.3 and is what `prompts/plan-bootstrap.md:31-51`
instructs Claude to emit.

```markdown
# Mission: <one-line>

## Sprint 1: <title> {#sprint-1}
**Goal:** <statement>
**Work front:** <front>

### Epic 1.1: <title> {#sprint-1.epic-1}
**Goal:** <statement>

#### Task 1.1.1: <title> {#sprint-1.epic-1.task-1}
**Goal:** <statement>
**Acceptance criteria:**
- <criterion>
- <criterion>
**Review:** [<reviewer-1>, <reviewer-2>]
**Validate:** `<command>`
**Work front:** <front>

#### Task 1.1.2: ...
```

Conventions:

- The `{#id}` anchor on each heading **must match** the corresponding
  `tree.json` node `id`. This is what makes the round-trip work ‚Äî a user
  reading `plan.md` can search for a task id and find its `tree.json` node.
- Heading depth maps directly to `type`: `##` ‚Üí sprint, `###` ‚Üí epic,
  `####` ‚Üí task.
- `**Acceptance criteria:**` is a bulleted list immediately under the task
  heading; each bullet is a single criterion. Order is significant ‚Äî bullet
  index N corresponds to `acceptance_criteria[N]` and to `<evidence
  criterion="N" />` references.
- `**Review:**` is a comma-separated list inside `[ ... ]`. Empty list
  (`**Review:** []` or omit the line) means no review-gate.
- `**Validate:**` wraps the command in backticks. Omit the line if there is
  no automation.
- `**Work front:**` is a single label or absent.

The Markdown is for browsing; the engine never parses it directly. The
authoritative mapping is in the agent's hands when `/goal:plan` writes both
files (`prompts/plan-bootstrap.md:27-55`).

## 5. Round-trip rules

1. `/goal:plan` writes both `plan.md` and `tree.json` together
   (`prompts/plan-bootstrap.md:27-55`). Initial state is
   `lifecycle="draft"`.
2. While `lifecycle ‚àà {draft, approved}`, the user may edit `plan.md` and
   `tree.json` directly. Any edit must keep them in sync ‚Äî there is no
   automated reconciliation.
3. `/goal:approve-plan` validates `tree.json` (see ¬ß6) and on success stamps
   `tree.approved_at` and transitions state to `approved`
   (`engine/approve-plan-cli.mjs:94-118`). It refuses on lifecycles other
   than `draft`/`approved` (the C-1 lifecycle gate at
   `engine/approve-plan-cli.mjs:82-92`).
4. After `lifecycle="approved"`, the tree is locked. `/goal:start`
   transitions to `pursuing`. From here on, the engine mutates only
   per-node runtime fields (`status`, `evidence`, `review_attempts`,
   `notes`, `blocker_reason`) ‚Äî never `id`, `type`, `acceptance_criteria`,
   `review`, `validate`, or `children`.
5. Direct hand-edits to `tree.json` while `lifecycle="pursuing"` are not
   prevented, but on the next load the engine will re-validate the schema.
   Schema-violating edits cause the file to be backed up to
   `tree.json.broken-<ts>-<seq>` and `loadTree` returns `null`
   (`engine/state.mjs:155-167`); the Stop hook then silently exits
   (`engine/stop-hook.mjs:117-118`). This is the "I-6" defense ‚Äî see
   [ANTI-PATTERNS.md](./ANTI-PATTERNS.md).

## 6. What `/goal:plan` does (high level)

The slash command loads `prompts/plan-bootstrap.md` and instructs Claude to:

1. Genuinely study the project ‚Äî read README, package manifests, `git log`,
   `.claude/skills/`, `.claude/agents/`. (`prompts/plan-bootstrap.md:7-9`.)
2. Identify the stack and propose stack-appropriate `validate` commands.
3. Survey installed subagents and propose project-appropriate `review`
   entries.
4. Decompose the mission into Sprint ‚Üí Epic ‚Üí Task with explicit
   acceptance criteria per task.
5. Apply work-front labels.
6. Write both `plan.md` and `tree.json` to `.claude/goals/active/`.
7. Print a summary of counts, fronts, reviewers, and a suggested budget for
   `/goal:start` (`prompts/plan-bootstrap.md:58-65`).

The engine is intentionally not involved here ‚Äî `/goal:plan` is a Claude-side
authoring task. The engine only enters when `/goal:approve-plan` runs the
validator.

## 7. What `/goal:approve-plan` validates

The validator is `engine/validate-plan.mjs:49-86`. Steps in order:

1. **zod schema check** (`engine/validate-plan.mjs:53-58`). Calls
   `GoalTreeSchema.parse(tree)`. Fails fast on any shape error: missing
   field, wrong type, empty `id`/`title`/`goal`, task with empty
   `acceptance_criteria`, etc. The empty-criteria-on-task constraint is
   caught here via `engine/state.mjs:54-57`.
2. **Placeholder scan** (`engine/validate-plan.mjs:60-70`). Recursive
   descent. For every node's `title`, `goal`, and each
   `acceptance_criteria` entry, the regex
   `/\b(TBD|TODO|FIXME|XXX)\b|\?{3,}/i` (`engine/validate-plan.mjs:47`)
   must NOT match. Any hit is an error, blocks approval. The four
   alphabetic tokens are word-bounded; `???` (three or more `?`s) is
   matched unanchored.
3. **Reviewer availability** (`engine/validate-plan.mjs:74-80`, optional).
   When the CLI passes `availableReviewers` (the default ‚Äî see
   `engine/approve-plan-cli.mjs:75`), every `review[]` entry not present in
   the discovered reviewer set produces a **warning**, not an error.
   Warnings are printed but do not block approval. The discovery walks
   `~/.claude/{skills,agents}/` and `<cwd>/.claude/{skills,agents}/`
   (`engine/approve-plan-cli.mjs:42-49`).

On success: `tree.approved_at` is stamped, `state.lifecycle` is transitioned
to `approved` (or a fresh state.json is created if none existed yet ‚Äî
`engine/approve-plan-cli.mjs:97-118`), and a `plan-approved` history event is
appended (`engine/approve-plan-cli.mjs:119-125`).

On failure: the validator returns errors; the CLI prints them and exits 1
(`engine/approve-plan-cli.mjs:139-145`). No state mutation occurs.
